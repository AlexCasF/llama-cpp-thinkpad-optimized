# Q4/Q4 128K benchmark — 2026-07-14

## Objective

Find the fastest usable configuration for the verified
`Huihui-Qwen3.6-35B-A3B-Claude-4.7-Opus-abliterated-ggml-model-Q4_K.gguf`
on the ThinkPad Ultra 5 228V / Arc 130V / 32 GB unified memory machine,
while two VMware guests were running.

The production target was changed from 256K to 128K context. All benchmark
cases used q4 KV cache, because host and iGPU memory are shared.

## Recipe findings

| Recipe idea | Decision |
|---|---|
| `--n-gpu-layers 99` | Kept in every run. |
| `--n-cpu-moe` | Tuned directly; 36 was the safest and fastest tested value. |
| Decode threads | Tested 4, 5, 6, 7, and 8. |
| Physical ubatch | Tested 256, 512, 1024, 1536, and 2048. |
| Logical batch | Compared 1024 and 2048. |
| Flash attention | Compared `auto` and `on`. |
| mmap | Kept on. No-mmap was excluded because it duplicates model residency and threatens the VM/RAM budget. |
| mlock | Excluded; it improves paging stability, not fresh-run speed, and can starve the host. |
| Cache reuse 256 | Kept for server configuration; it does not change raw llama-bench decode speed. |
| TurboQuant | Not available in the pinned official build. |
| Speculative decoding | Excluded; the transcript reports a regression for this hybrid MoE/SSM architecture. |
| REAP | Excluded; it requires a separately pruned model, not this GGUF. |

## Benchmark method

- Runtime: official llama.cpp Windows SYCL build `b9986`, commit `91c631b21`.
- Backend: Intel Arc SYCL, `--n-gpu-layers 99`.
- KV: `q4_0` keys and `q4_0` values.
- mmap: enabled.
- Workload: `-p 2048 -n 128`; PP is prompt processing and TG is token generation.
- Two VMware processes remained active. Ollama had no model loaded.
- Most cases were one repetition; the first MoE sweep and finalists used repeated runs.
- A multi-case sweep was stopped when it reduced free RAM to about 1 GiB. Subsequent cases were run one at a time so model memory was released between cases.

## Deployment confirmation

The leading production shape was started with:

```text
ctx 131072, CPU MoE 36, threads 6, threads-batch 7,
batch 1024, ubatch 1024, q4_0/q4_0, flash-attn on, mmap on
```

The server became healthy with two VMware guests active and left about
13.8 GiB free system RAM after loading. A short live completion measured
18.35 generated tokens/s. This request is not directly comparable to the
llama-bench 128-token TG rows, but confirms the selected 128K configuration
loads and runs in the intended environment.

## Conclusion

The best configuration to ship as the 128K production default is:

```text
N_CPU_MOE=36
THREADS=6
THREADS_BATCH=7
BATCH_SIZE=1024
UBATCH_SIZE=1024
CACHE_TYPE_K=q4_0
CACHE_TYPE_V=q4_0
FLASH_ATTN=on
LLAMA_NO_MMAP=0
LLAMA_MLOCK=0
```

The single-run t7 result was slightly faster, but the repeated t7 run was
unstable under sustained VMware load. t6 gave the more defensible production
choice and the live server confirmation reached 18.35 tok/s. Keep `N_CPU_MOE=36`:
moving more expert layers to the iGPU increased host pressure and reduced
throughput in this environment.
